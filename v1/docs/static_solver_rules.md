# Static Solver Rules

这份文档定义 `v1` 当前静态谜题版本的求解规则、UI 与求解器共用的判断边界，以及后续扩展到新规则时的改动范围。

## 1. 当前阶段的目标

`v1` 当前的求解器目标是：

- 只处理静态谜题
- 以最少操作步数为目标
- 支持内部终点和边缘终点
- 支持 `full` / `partial` 两种终点判定
- 与编辑器验证模式共用同一套合法移动语义

这里的“静态谜题”指：

- 玩家做一步
- 棋盘不会自动继续变化
- 没有风、重力、传送带自动推进、周期变化这类环境结算

## 2. UI 红框 / 绿框与求解器的关系

编辑器里的预览框是 UI 表现，不是求解器本体。

- 绿框：当前拖拽落点合法
- 红框：当前拖拽落点不合法

求解器不需要复制“画框”这件事，但必须复用同一套合法性判断。

推荐边界如下：

- UI 层：只负责调用规则判断并显示红/绿预览
- 规则层：负责判断“这一步能不能走”
- 求解器：负责枚举合法动作并搜索最优解

在当前实现里，这套关系已经基本成形：

- UI 侧的移动判断集中在 `foundation/ui/app.js`
- 运行时终点判断集中在 `foundation/core/runtime.js`
- 内置静态 BFS 集中在 `foundation/solver/adapter.js`

后续如果把求解器迁到 Java，Java 应该复刻的是“合法移动规则”和“终点判定规则”，而不是复刻红框/绿框 UI。

## 3. 当前静态规则的核心元素

### 3.1 格子标签

当前基础静态规则包包含这些格子标签：

- `free`
- `blocked`
- `horizontal`
- `vertical`
- `target-lane`
- `block-lane`

含义如下：

- `free`
  普通可通过格
- `blocked`
  禁止进入
- `horizontal`
  普通运动规则格，物体在这里优先遵守横向运动
- `vertical`
  普通运动规则格，物体在这里优先遵守纵向运动
- `target-lane`
  目标物轨道
- `block-lane`
  普通障碍物轨道

一个格子允许同时拥有多个标签。

### 3.2 物体类型

当前静态版本的基础物体有：

- `target`
- `block`
- `fixed`

含义如下：

- `target`
  目标物，可以参与终点判定
- `block`
  普通可移动障碍物
- `fixed`
  固定障碍物，不能移动

### 3.3 区域类型

当前求解规则依赖的区域主要是：

- `spawn`
- `goal`

其中 `goal` 同时支持：

- 内部矩形终点
- 边缘终点

### 3.4 终点模式

当前支持两种终点判定：

- `full`
- `partial`

含义如下：

- `full`
  目标物必须完整进入终点区域
- `partial`
  目标物只要有一部分进入终点区域即可

对于边缘终点，这个差异尤其重要。

例如：

- `2x2` 目标物进入右侧 `1x2` 边缘终点
- `full` 下不成立
- `partial` 下成立

## 4. 当前移动规则

### 4.1 基本原则

验证模式和求解器都应遵守同一组移动规则：

1. 物体不能与其他物体重叠
2. 物体不能进入 `blocked`
3. 物体不能越出棋盘，除非进入匹配的边缘终点
4. `fixed` 不能移动
5. 目标物和障碍物都可以被求解器移动

### 4.2 设计模式与验证模式

当前系统区分两种编辑体验：

- 设计模式
  允许自由摆放，忽略大多数运动规则
- 验证模式
  严格按规则移动

求解器应该对齐的是“验证模式”，不是“设计模式”。

### 4.3 轨道优先级

当前静态规则的轨道优先级是：

1. 普通运动规则格优先
   包括 `horizontal` / `vertical`
2. 角色专属轨道其次
   包括 `target-lane` / `block-lane`
3. 未标记格最后

具体解释：

- 如果物体当前进入了 `horizontal` 或 `vertical` 轨道，则必须优先服从这些普通运动规则
- `target-lane` 主要约束目标物
- `block-lane` 主要约束障碍物
- 重合格允许多标签共存

对于目标物：

- 当格子同时有 `target-lane` 和普通运动规则时
- 普通运动规则优先

对于障碍物：

- 当格子同时有 `block-lane` 和普通运动规则时
- 普通运动规则优先

### 4.4 边缘终点的特殊规则

边缘终点不是普通棋盘外区域，而是“只对目标物开放的合法延伸区域”。

当前边缘终点规则如下：

- 只有 `goal` 角色的边缘区域可被当作棋盘外合法区域
- 只有目标物可以进入边缘终点
- 其他边缘区域不是合法移动空间
- 目标物进入边缘终点这一步，不再被普通轨道规则错误阻挡

这条规则是为了保证下面这种场景成立：

- 目标物沿着目标轨道移动
- 轨道直接连接到边缘终点
- 最后一步可以进入终点区，而不会因为“终点格本身没有轨道标签”被卡住

## 5. 当前求解器应该复用的判断

当前静态求解器真正应该复用的是以下三类判断：

### 5.1 放置是否合法

推荐接口语义：

- `canOccupyFootprint(state, piece, row, col, width, height)`

职责：

- 边界检查
- 重叠检查
- `blocked` 检查
- 是否允许进入边缘终点

### 5.2 移动是否合法

推荐接口语义：

- `canMovePiece(state, piece, row, col, fromRow, fromCol)`

职责：

- 复用放置合法性
- 检查方向是否符合普通运动规则
- 检查是否符合 `target-lane` / `block-lane`
- 放行目标物进入边缘终点的最后一步

### 5.3 是否已经到达终点

推荐接口语义：

- `evaluateGoals(state)`

职责：

- 统一处理内部终点与边缘终点
- 统一处理 `full` / `partial`

### 5.4 谜题配置是否有效

推荐接口语义：

- `validatePuzzle(state, rulePack)`

职责：

- 是否存在目标物
- 是否存在终点
- 基础结构是否合法

这个阶段的 `validatePuzzle` 不负责“是否可解”，只负责“配置是否合法”。

## 6. 当前静态求解器的搜索模型

当前静态求解器可以按下面的模型实现：

- 状态：棋盘 + 物体 + 区域
- 动作：移动某个可移动物体
- 目标：满足 `evaluateGoals(state)`
- 去重：序列化状态
- 搜索：BFS，目标是最少操作步数

这里的关键原则是：

- 求解器不直接操作 DOM
- 求解器不关心红框绿框
- 求解器只关心状态、动作、合法性判断、终点判断

## 7. 后续加入新规则时是否需要大改

结论是：

- 会扩展
- 但不应该推翻重写

前提是继续保持下面这个边界：

- 规则层
  负责合法性判断和状态转移
- 求解器
  负责搜索
- UI
  负责交互和表现

### 7.1 加入开关

如果后面加入开关，通常会新增：

- 运行时状态
- 开关触发结果
- 某些格子/区域/门的启用状态

影响范围通常是：

- `state`
- `canMovePiece`
- `evaluateGoals`
- 状态序列化

这属于“小到中等改动”，但不应该推翻整个搜索框架。

### 7.2 加入传送带

如果后面加入传送带，要先区分两类：

- 静态传送
  玩家进入某格后立刻传送一次
- 自动结算型传送带
  玩家走一步后，环境继续自动推进

第一类通常仍可放在当前静态模型里。

第二类会把求解器从纯静态推进到“环境结算”模式，也就是：

- 玩家动作
- 环境自动推进
- 得到稳定状态

这时需要扩展：

- 状态转移逻辑
- 环境结算接口
- 循环检测

但搜索框架本身依然可以保留。

## 8. 建议的分层边界

为了后续接 Node、Java、AI，不建议把规则判断写散在 UI 事件里。

建议继续收束成下面三层：

### 8.1 基础规则层

只由维护者修改：

- 状态模型
- 合法性判断
- 终点判断
- 求解器接口

### 8.2 规则包层

允许未来继续扩展：

- 格子标签
- 物体类型
- 区域类型
- 胜利条件配置

### 8.3 UI / 桥接层

负责：

- 编辑器交互
- 红框/绿框预览
- 与 Node/Java/AI 通信

## 9. 对 Node + Java 的意义

这份规则整理出来以后，后面可以按这个方式衔接：

- 前端 TypeScript / JS
  负责编辑器和预览
- Node
  负责桥接层
- Java
  负责求解器

其中：

- UI 红框/绿框继续留在前端
- `canMovePiece` / `evaluateGoals` 的语义由 Java 复刻
- Node 只负责传输 `PuzzleSpec` 和 `Solution`

## 10. 当前阶段的结论

当前静态谜题版本已经满足：

- 可以正式开始写独立求解器
- 可以用 Node 作为桥接层
- 可以把 Java 作为高性能求解核心

但后续新增机制时，应该坚持这个原则：

UI 表现可以变，求解器实现可以换语言，但规则语义必须只维护一份。
