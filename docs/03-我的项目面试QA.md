# Puzzle V1 Agent 项目面试问答精编

面向岗位：`Agent 开发岗 / AI 应用工程师 / 多智能体系统工程师`

适用项目：`Puzzle V1`，一个把谜题编辑器、规则包、求解器、AI 工作台、多 Agent 编排、工具调用、RAG 与 Memory 串起来的系统。

建议使用方式：
- 先熟背“项目 3 分钟介绍”和前 10 题。
- 再按岗位重点补“多 Agent / Function Calling / RAG / Memory / 求解器 / 技术债”。
- 每题建议准备：
  1. 30 秒短答
  2. 90 秒展开版
  3. 1 个真实代码或架构细节

## 一、项目整体设计（8 题）

### Q1. 你的项目是做什么的？核心价值是什么？
**参考答案：**  
我的项目表面上是一个谜题编辑器，但我把它做成了一个受控的 Agent 系统。用户不仅能编辑棋盘、物体和终点区，还能让 AI 分析当前关卡、给出修改建议、后台自动尝试改关，并且把结果用确定性求解器重新验证。项目核心价值不是“接了个聊天接口”，而是把 LLM 放进了一个可验证、可回放、可解释的闭环里。系统把规则定义、求解逻辑、UI 交互和 AI 编排拆层，既能支持本地原型，也能逐步演进成正式的 Agent 应用。

**面试官想听的点：**  
能把项目定位从“前端工具”上升到“Agent 系统”；讲清楚用户价值和技术价值。

**常见踩坑：**  
只讲页面功能和按钮，不讲系统闭环、可控性和工程设计。

---

### Q2. 为什么这个项目适合做成 Agent，而不是普通规则系统或聊天机器人？
**参考答案：**  
因为这个任务既不是纯规则问题，也不是纯聊天问题。用户会用自然语言提出模糊目标，比如“把这个关卡改难一点，但不要扩大棋盘”，这一步需要 LLM 做意图理解和策略生成；但修改后的关卡是否合法、是否可解、最短步数是多少，这些又必须由确定性系统完成。所以我把它设计成 Agent：模型负责理解意图和生成编辑策略，工具负责验证、求解和打分。这样比纯规则系统更灵活，比普通聊天机器人更可靠。

**面试官想听的点：**  
知道 LLM 和确定性程序分别适合做什么；能说明 Agent 的边界。

**常见踩坑：**  
把 Agent 简化成“能聊天”；或者反过来说“全都用规则就行”，忽略用户请求的自然语言模糊性。

---

### Q3. 这个项目为什么要分成前端编辑层、规则包层、求解器层、AI 编排层？
**参考答案：**  
我做分层的核心原因是降低耦合。前端编辑层负责棋盘编辑、拖拽、导入导出、回放和 AI 面板；规则包层定义 cell tag、piece、zone、goalMode 等语义；求解器层只关心状态搜索；AI 编排层只关心用户意图、工具调用和多 Agent 协作。这样做的好处是，规则一旦稳定，前端和求解器都可以共享同一套契约；同时求解器可以从内置 JS 逐步升级到 Node/Java，而不影响 UI；AI 也只能通过受控接口接入，而不是直接修改基础层。

**面试官想听的点：**  
分层目的、解耦收益、未来扩展路径。

**常见踩坑：**  
只说“为了代码整洁”，但说不清每一层分别负责什么。

---

### Q4. 你的项目里“单一真相源”是什么？
**参考答案：**  
我的单一真相源不是某个 UI 状态，而是谜题规格本身，也就是包含 `board / pieces / zones / meta` 的结构化关卡对象，再加上规则包定义出来的规则语义。前端、Node bridge、Java solver、AI tools 都围绕这个结构化 puzzle spec 工作。这样一来，不同组件虽然实现方式不同，但输入输出契约是一致的。这个设计的价值在于，任何一层都不能定义一套“只有自己知道”的规则，否则就会产生语义漂移。

**面试官想听的点：**  
知道“单一真相源”是数据契约和规则契约，不只是某个变量。

**常见踩坑：**  
把单一真相源等同于“前端 state”或者“数据库某张表”，缺乏系统视角。

---

### Q5. 如果让你用 3 分钟介绍这个项目，你怎么讲？
**参考答案：**  
我会按 STAR 来讲。Situation 是一开始它只是一个谜题编辑器，但如果要支持自动分析、自动改关和 AI 协作，就会遇到规则耦合、求解器不可扩展、AI 直接改 JSON 风险太高等问题。Task 是把它升级成一个受控的 Agent 系统。Action 是四部分：第一，按 UI、规则包、求解器、AI 编排分层；第二，设计 planner、generator、critic、controller 多 Agent workflow；第三，用 function calling 加原子编辑约束模型修改；第四，用 RAG 和 session memory 让系统支持连续多轮工作。Result 是这个系统已经能在本地工作台里做连续对话、后台修改任务、确定性验证和求解回放。

**面试官想听的点：**  
表达能力、结构化总结能力、结果导向。

**常见踩坑：**  
讲太散，按文件夹介绍，而不是按问题、方案、结果介绍。

---

### Q6. 这个项目里最有亮点的设计是什么？
**参考答案：**  
我认为最有亮点的是“用受控工具链把 LLM 关进闭环”。我没有让模型直接返回整份新关卡，而是要求它只能通过 function calling 调用工具，并且工具只能接收原子编辑操作。编辑之后还必须通过 validate、solve、score 这些确定性步骤重新验证。这意味着模型可以提出策略，但不能直接越权改状态。这个设计既保留了 LLM 的灵活性，又把幻觉和越权的风险压在了系统边界之内。

**面试官想听的点：**  
知道什么是“Agent 可控性”；能讲出自己的设计哲学。

**常见踩坑：**  
把亮点说成“用了多 Agent”或“用了 LangGraph 风格”，缺少真正的工程落点。

---

### Q7. 这个项目最难的地方是什么？
**参考答案：**  
最难的地方不是某个算法，而是把“自然语言意图”转换成“受限编辑动作”，再把结果放回“可验证的谜题系统”里。也就是说，前半段是 AI 问题，后半段是工程问题。如果只做聊天，问题简单很多；如果只做求解器，也简单很多。真正困难的是中间这层把 LLM、工具、规则和搜索系统接起来的过程。我在这里的核心做法是结构化 brief、工具白名单、原子编辑、确定性校验和 controller 决策。

**面试官想听的点：**  
能识别系统中真正的复杂点；不是只挑表面上最“炫”的部分讲。

**常见踩坑：**  
回答成“最难的是调 prompt”。

---

### Q8. 这个项目如果继续做产品化，你会优先做什么？
**参考答案：**  
我会优先做三件事。第一，统一规则回归，把前端、Node、Java 跑在同一批 golden cases 上，解决语义漂移风险。第二，把后台 modify job 从单机线程模型升级成持久化任务队列，提高恢复能力和可观测性。第三，继续增强 memory 和评估体系，让多轮编辑和 AI 修改结果更稳定。也就是说，我不会先追求“更多花哨 agent”，而是优先补系统可靠性和演进基础设施。

**面试官想听的点：**  
知道什么是产品化优先级；不是只会继续加功能。

**常见踩坑：**  
回答成“接更多模型”或者“加更多前端效果”。

## 二、Agent 架构与多智能体协同（8 题）

### Q9. 为什么要用多 Agent，而不是一个大 Agent 一把梭？
**参考答案：**  
因为这个任务天然包含不同阶段，而且每个阶段的目标函数不一样。planner 负责理解用户需求并产出结构化设计 brief；generator 负责真的去改关卡；critic 负责审查结果是否符合 brief、是否合法、是否可解；controller 负责接受、重试或者返回最佳候选。单 Agent 把这些职责混在一起，很容易出现自我确认偏差，也很难定位问题到底出在意图理解、执行还是评估阶段。多 Agent 的代价是成本和延迟更高，但收益是职责清晰、失败隔离和可解释性更强。

**面试官想听的点：**  
知道多 Agent 的收益和代价；不是为了“高级”而多 Agent。

**常见踩坑：**  
只说“多 Agent 更专业”，不讲为什么单 Agent 不够。

---

### Q10. 你的多 Agent 是怎么分工的？
**参考答案：**  
我的分工是 planner、generator、critic、controller。planner 主要负责把自然语言请求编译成 `DesignBrief`，包括 intent type、hard constraints、soft targets、generation spec、review rubric。generator 根据 brief 通过工具调用和原子编辑来生成候选关卡。critic 读取同一份 brief，用确定性工具和模型联合评估 candidate。controller 负责多轮尝试、决定是否接受以及在没达到完美结果时返回 best-effort candidate。这样每个角色都有清晰输入输出，不需要在一个 prompt 里做所有事。

**面试官想听的点：**  
角色定义清楚；知道中间契约的重要性。

**常见踩坑：**  
说成“planner 管计划，generator 管生成”这种空话，没有具体结构化对象和决策流。

---

### Q11. 多 Agent 之间怎么协作？靠共享上下文还是消息传递？
**参考答案：**  
我这里优先用了结构化对象协作，而不是自由文本消息传递。最重要的中间契约是 `DesignBrief` 和 candidate level。planner 输出 brief，generator 和 critic 都读同一份 brief；generator 产出 updated level，critic 和 controller 再围绕这个 level 评估。这样协作的核心不是“大家互相聊天”，而是“通过结构化对象交接”。相比自由文本，这样更稳定，更利于调试，也更适合在代码里做 schema 校验。

**面试官想听的点：**  
知道多 Agent 协作最关键的是“中间契约”，不是 agent 数量。

**常见踩坑：**  
默认把多 Agent 想成自然语言转发链条。

---

### Q12. 为什么还需要一个 controller，不能让 critic 直接决定吗？
**参考答案：**  
critic 是评估者，不应该同时负责流程控制。controller 的作用是把 deterministic constraints、critic score 和是否有实际编辑这些因素统一起来，决定 accept / retry / best candidate。这样做的好处是把“质量评估”与“流程决策”分开。critic 只需要说这个 candidate 好不好、哪里有问题；controller 再决定是否重试、是否停止、是否带 warning 返回。这样可扩展性更好，也更符合单一职责。

**面试官想听的点：**  
职责边界；知道为什么流程控制要独立。

**常见踩坑：**  
让 critic 同时评审、改错、决定重试，导致角色混乱。

---

### Q13. 单 Agent 和多 Agent 你怎么选？
**参考答案：**  
我主要看三个维度。第一，任务是不是天然分阶段；第二，是否需要生成和评审分离；第三，失败后是否需要定位到具体环节。如果任务只是单步问答、简单工具调用或结构很短，我会优先单 Agent，因为成本和延迟都更低。只有在任务包含规划、执行、审查这类目标函数明显不同的阶段时，我才会用多 Agent。我的项目属于后者，因为它同时涉及意图理解、编辑执行和结果审查。

**面试官想听的点：**  
有选择标准；不是“永远多 Agent 更好”。

**常见踩坑：**  
不谈 cost/latency/complexity，只强调能力上限。

---

### Q14. 如果 generator 改坏了关卡，系统怎么恢复？
**参考答案：**  
第一层，generator 只能通过原子编辑调用 `apply_level_edits`，这天然就限制了破坏范围。第二层，编辑后必须做 validate、solve 和 score，不通过就不会被当作 accepted candidate。第三层，controller 会根据 constraint evaluation 和 critic report 决定是否重试。也就是说，系统默认承认 generator 可能犯错，但不会让错误直接落地成最终结果。这种设计本质上是一种带审查的候选生成，而不是一次性覆盖。

**面试官想听的点：**  
失败恢复、候选机制、系统保护边界。

**常见踩坑：**  
把“generator 可能出错”说成问题，而不是用系统设计化解它。

---

### Q15. 这个多 Agent workflow 和 LangGraph 这类编排框架是什么关系？
**参考答案：**  
我当前的实现思路已经是图编排风格了：有明确节点、状态对象和条件流转，只是没有把所有节点都完全外部图配置化。我的节点包括 intent compilation、chat logic、modify agent、rule refactor 等，整体是一个有状态图语义的系统。之所以没有把一切都抽成更重的图编排平台，是因为当前规模下直接用 Python 代码更容易读、也更容易联调。如果后面节点更多、需要 checkpoint、人工干预或更强的图可视化，再进一步平台化是自然演进方向。

**面试官想听的点：**  
能区分“图思维”和“图框架”；有演进路线意识。

**常见踩坑：**  
把没用某个框架理解成“没有工作流”；或者反过来生搬硬套框架名词。

---

### Q16. 你的多 Agent 设计里，最重要的工程原则是什么？
**参考答案：**  
最重要的是“结构化中间契约 + 受控工具边界”。如果只有多 Agent 但没有统一契约，系统会变成很多 prompt 的串联，调试会非常痛苦；如果只有工具但没有角色分工，系统又容易变成一个大一统 agent。我的做法是，让 planner 产出结构化 brief，让 generator 和 critic 都以这个 brief 为准，再通过工具调用访问真实世界。这样角色边界、状态边界和执行边界都比较清楚。

**面试官想听的点：**  
知道多 Agent 工程化真正难点在哪。

**常见踩坑：**  
把重点放在“角色命名”而不是“契约设计”。

## 三、Function Calling、ReAct、原子编辑、Skill、MCP（7 题）

### Q17. 你项目里的 function calling 解决了什么问题？
**参考答案：**  
它解决的是“模型和确定性系统之间如何交互”的问题。模型擅长理解用户意图和生成策略，但合法性、可解性、难度评估、状态修改这些都不应该由模型自由决定。我通过 function calling 把 `get_current_level`、`validate_level`、`solve_level`、`score_level`、`diff_current_level`、`apply_level_edits` 这些确定性能力暴露给 agent，并且按角色限制工具集合。这样模型只能通过受控接口读写系统状态，而不是直接输出一份未经验证的最终答案。

**面试官想听的点：**  
知道 function calling 的本质是“受控执行接口”。

**常见踩坑：**  
把 function calling 说成“模型输出 JSON”。

---

### Q18. 什么是原子编辑？为什么它在你的项目里很重要？
**参考答案：**  
原子编辑就是把一次修改拆成最小、明确、可验证的操作，而不是让模型一次性重写整份关卡对象。在我这个项目里，像 `add_piece`、`update_piece`、`set_cell_tags`、`add_zone` 这些就是原子操作。它的重要性在于四点：第一，可控，知道到底改了什么；第二，可验证，每次编辑后都能立即 validate 和 solve；第三，可回滚，问题能定位到具体操作；第四，可审计，前端也能展示 diff。它和 function calling 结合后，相当于给模型套了两层限制：既不能直接改状态，又不能一次性大改。

**面试官想听的点：**  
理解“最小动作”的价值；不是只会讲概念。

**常见踩坑：**  
把原子编辑理解成“把 JSON 切小一点”。

---

### Q19. 你理解的 ReAct 是什么？你项目里有吗？
**参考答案：**  
ReAct 的核心是 Thought / Action / Observation 的交替循环。我的项目不是把思维链原样暴露出来，而是做了工程化版 ReAct：agent 拿到系统 prompt 和结构化 payload 后，如果模型产生 tool calls，系统就执行工具，再把 tool result 作为 observation 喂回去，循环若干轮，直到模型产出最终 JSON 或文本结果。这在 `run_tool_agent` 里体现得比较清楚。也就是说，我没有把 ReAct 当成论文术语，而是把它落成了“工具调用闭环”。

**面试官想听的点：**  
知道 ReAct 不是 prompt 名词，而是执行范式。

**常见踩坑：**  
说“我没显式写 Thought，所以没有 ReAct”。

---

### Q20. 如果面试官问你 skill，你怎么结合项目回答？
**参考答案：**  
我会把 skill 理解成一组稳定、可复用的能力包，而不是单纯的一段 prompt。一个 skill 通常包括角色定义、允许的工具、输入输出契约、提示模板和执行流程。从这个角度看，我的项目虽然没有单独做 skill registry，但已经具备 skill 化能力了。比如“关卡分析 skill”“关卡修改 skill”“候选审查 skill”就是不同的 prompt、不同的工具权限、不同的输出 schema。后续如果做产品化，我完全可以把这些能力进一步抽象成显式 skill 层。

**面试官想听的点：**  
对 skill 的理解有工程边界；能落回项目。

**常见踩坑：**  
把 skill 说成“某个大神写的 prompt”。

---

### Q21. 如果面试官问你 MCP，你怎么回答比较稳？
**参考答案：**  
我会先明确说，我项目当前不是正式 MCP 协议实现，但架构已经具备 MCP-compatible 的能力拆分。因为我已经把规则文档、案例库、当前关卡状态、validate/solve/score/apply edits 这些能力拆成了独立资源和工具。MCP 的价值在于把这些能力用统一协议暴露出来，让模型通过标准方式访问 resource 和 tool。如果后续接 MCP，我会优先把规则文档、案例库和当前关卡暴露成 resources，把 validate、solve、score、apply edits 暴露成 tools。

**面试官想听的点：**  
诚实；知道 MCP 的定位；能说改造路径。

**常见踩坑：**  
没做过也硬说做过，或者完全不知道 MCP 是“协议层”。

---

### Q22. 你怎么限制模型乱调工具或者越权调工具？
**参考答案：**  
我这里做了三层限制。第一层，按角色限制允许的工具集合，比如 planner 不能直接做 destructive edits，critic 只有只读工具。第二层，工具本身也做了能力约束，比如 `apply_level_edits` 只接收支持的原子操作，不接受任意全量覆盖。第三层，工具执行结果还要再走 validate、solve、score 这些后验检查。也就是说，模型不是没有行动能力，而是行动能力被放进了严格的边界里。

**面试官想听的点：**  
权限边界、最小权限原则、双层校验。

**常见踩坑：**  
只在 prompt 里写“不允许”，没有系统级约束。

---

### Q23. Tool trace 的作用是什么？
**参考答案：**  
tool trace 是我保留下来的可观测执行轨迹。它至少有四个作用：第一，调试，知道 agent 每轮调了什么工具、传了什么参数、拿到了什么结果；第二，审计，知道最终答案是不是建立在真实工具返回之上；第三，前端可解释性，可以向用户展示修改过程；第四，后续评估和数据集建设，可以分析哪些工具调用模式成功率更高。它本质上是 Agent 系统的“可回放日志”。

**面试官想听的点：**  
知道可观测性在 Agent 系统里有多重要。

**常见踩坑：**  
把 trace 只当 debug 临时信息，不当系统资产。

---

## 四、RAG 与 Memory（6 题）

### Q24. 你项目里为什么要做 RAG？检索的是什么？
**参考答案：**  
我做 RAG 的目的是给 agent 提供 grounding，减少它在规则解释和设计审查上的主观发挥。我的项目里有两类 RAG：一类是规则文档 RAG，检索规则定义、边界和 solver behavior；另一类是案例库 RAG，检索最小正例、反例、无解例和非法例。这两类上下文主要给 planner 和 critic 用，因为它们最需要“知道规则”和“知道历史模式”。generator 也会用到案例检索，但更多是作为 seed pattern 参考。

**面试官想听的点：**  
知道 RAG 的目标是 grounding，不是盲目接知识库。

**常见踩坑：**  
把 RAG 说成“向量库就是高级记忆”。

---

### Q25. 规则文档 RAG 和案例库 RAG 为什么要分开？
**参考答案：**  
因为它们提供的是两种不同类型的知识。规则文档回答的是“系统允许什么、不允许什么”；案例库提供的是“以前哪些模式是有效的、哪些是反例”。前者更像约束知识，后者更像经验知识。把它们混在一个检索池里，容易让 agent 把案例当规则，或者把规则当模式模板。分开之后，planner 可以更好地把规则 grounding 和案例示例同时纳入 brief，critic 也能分别用“规则边界”和“历史模式”做评审。

**面试官想听的点：**  
能区分不同知识源的角色。

**常见踩坑：**  
所有文本都扔进一个库里，认为检索结果越多越好。

---

### Q26. 你的 memory 怎么设计？为什么不是只存聊天记录？
**参考答案：**  
我把 memory 设计成任务状态 memory，而不是纯聊天记忆。除了 messages，我还保存 `working_level`、`analysis_summary`、`last_intent` 和 `suggestions`。原因是用户和 agent 之间不是一般问答，而是在持续改一个关卡。如果只存聊天记录，agent 很难稳定知道“当前正在修改哪个版本的关卡”“上一轮接受了什么方案”“这轮是不是在追问上次建议”。所以真正关键的是工作上下文，而不仅仅是语义对话。

**面试官想听的点：**  
知道 memory 应服务于任务，而不只是服务于对话连续性。

**常见踩坑：**  
把 memory 等同于把历史消息拼接到 prompt 里。

---

### Q27. Memory 会不会越来越脏？你怎么处理？
**参考答案：**  
会，所以 memory 设计的关键不是“记更多”，而是“保留下一步真正需要的状态”。我的做法是把 session 归一化成标题、预览、messages、working_level、analysis_summary、last_intent、suggestions 这几类字段，而不是把所有中间状态和所有推理结果都长期存下来。这样可以避免无关噪声一直累计。后续如果要继续增强，我会再加摘要压缩、重要事件提取和 relevance-based memory selection，而不是无限扩消息历史。

**面试官想听的点：**  
知道 memory 污染问题；有压缩和筛选意识。

**常见踩坑：**  
默认认为“上下文越长越好”。

---

### Q28. RAG 和 Memory 的边界怎么区分？
**参考答案：**  
我的区分方式是：RAG 负责共享知识，Memory 负责会话状态。规则文档和案例库属于任何用户都能复用的知识，所以它们是 RAG；当前 working level、last intent、上一轮建议这些只对当前会话有意义，所以它们是 memory。如果把案例库硬塞进 memory，会让每个会话都带着大量无关知识；如果把用户当前工作状态做成 RAG，又会丢掉强时序语义。两者应该互补，而不是互相替代。

**面试官想听的点：**  
边界清晰；不是把两者混为一谈。

**常见踩坑：**  
回答成“RAG 就是外部 memory”。

---

### Q29. 你的 session store 为什么同时支持 SQLite 和 MySQL？
**参考答案：**  
这是一个典型的演进式设计。SQLite 非常适合本地开发和单机 demo，部署简单、依赖少；MySQL 更适合多用户或服务化场景。所以我不是一开始就把 session store 做成复杂分布式系统，而是先支持一个本地好用的默认后端，再给出更正式的数据库后端选项。对 Agent 系统来说，这样的设计能兼顾开发效率和演进空间。

**面试官想听的点：**  
知道“默认简单 + 可升级”是一种合理工程策略。

**常见踩坑：**  
为了“企业级”一上来就把 demo 做得过重。

---

## 五、求解器、算法与搜索（8 题）

### Q30. 为什么你要单独做求解器，而不是让 LLM 直接判断关卡好不好？
**参考答案：**  
因为“关卡是否合法、是否可解、最少操作步数是多少”属于确定性问题，不应该交给 LLM 主观猜。LLM 可以辅助做设计建议，但不能代替求解器。我的系统里，validate、solve、score 都是确定性工具链的一部分，这也是为什么 generator 修改完关卡后必须重新求解和打分。这样用户看到的 AI 修改结果，至少是在规则和搜索层面站得住的。

**面试官想听的点：**  
确定性系统和概率模型的边界意识。

**常见踩坑：**  
认为“模型觉得像个好关卡”就够了。

---

### Q31. 为什么要用 Java 做正式求解器，而不是统一用 Python？
**参考答案：**  
因为我把系统按工作负载拆层了。Python 更适合做 AI orchestration、FastAPI、tool calling、RAG 和 memory；求解器是 CPU 密集型状态搜索，更适合放在一个强类型、性能更稳定、便于长期优化的语言里。Java 在状态建模、优先队列、去重和后续更复杂启发式优化上更适合作为正式路径。我不是说 Python 不能做 solver，而是说在这个项目里，语言统一不如职责匹配重要。

**面试官想听的点：**  
按工作负载选技术；知道语言选择是工程判断。

**常见踩坑：**  
说“Java 就一定比 Python 快”，但讲不出系统层面的原因。

---

### Q32. 为什么前端 baseline 用 BFS，而 Java 正式版做成 A* 风格搜索？
**参考答案：**  
BFS 很适合做第一版 baseline，因为它实现简单、规则验证清晰，而且在单位步长一致时天然能得到最少操作步数，所以非常适合先验证正确性。但 BFS 是无信息搜索，状态一大、障碍一多、规则更复杂时，frontier 会膨胀得很快。Java 正式版切到 A*-style informed search，是为了把领域知识纳入搜索顺序，比如目标距离、通道清理、阻挡惩罚等，让系统更快找到高质量解。这是从 correctness baseline 向 performance path 的自然演进。

**面试官想听的点：**  
baseline 与正式实现的分工；不是“推翻重来”。

**常见踩坑：**  
把 BFS 讲成“低级做法”，显得对前期设计没有整体认识。

---

### Q33. 为什么不用 DFS？
**参考答案：**  
DFS 在这个项目里不适合做正式求解器，因为它不保证最优解，而且很容易在深分支里浪费大量时间。我的编辑器场景需要的是稳定的最优步数解或接近最优的高质量解，还要支持回放和对难度的进一步分析。DFS 虽然实现简单、内存占用相对小，但在这类最短路/最优操作数问题里不是合适的默认选择。

**面试官想听的点：**  
知道算法选择和产品目标有关。

**常见踩坑：**  
机械背“DFS 不保证最短路”，但说不出为什么这对当前产品重要。

---

### Q34. 你的 Java 版 A* 一定严格最优吗？
**参考答案：**  
更准确地说，我现在的 Java 版是 A*-style informed search。它保留了 `g + h` 的结构，但 `h` 里加入了一些领域偏置和剪枝策略，目标是工程上更快找到高质量解，而不是严格按教材定义去证明 heuristic admissibility。如果面试官要求严格理论最优性，我会诚实说：要严格证明最优，需要更严格约束 heuristic；但当前实现的定位更偏工程性能优化。

**面试官想听的点：**  
诚实；知道理论最优性与工程启发式之间的差异。

**常见踩坑：**  
把所有 A* 都说成“严格最优”，经不起追问。

---

### Q35. 你做了哪些剪枝或启发式优化？
**参考答案：**  
我主要做了几类优化。第一，状态去重，用序列化 key 和 best cost 来避免重复展开。第二，反向移动抑制，减少无意义来回。第三，目标导向启发，比如更靠近目标区域的状态会更优先。第四，通道相关启发，清理关键通道会加分，堵住通道会惩罚。第五，明显无望状态的提前剪枝。它们共同目标是减少无效状态展开，而不是靠蛮力搜索。

**面试官想听的点：**  
优化不是抽象名词，能说出实际策略。

**常见踩坑：**  
只说“我做了剪枝”，但说不出剪了什么。

---

### Q36. 为什么还需要 Node bridge？不能前端直接调 Java 吗？
**参考答案：**  
浏览器不能直接可靠地管理本地 Java 进程，所以我在中间放了 Node bridge 这一层。它有三个作用：第一，桥接浏览器与本地求解器进程；第二，统一输入输出契约；第三，提供 fallback 策略，比如 Java 调用失败时回退到内置 JS solver。这样做的好处是前端只依赖一个稳定的 HTTP 接口，不需要关心底层到底是 Java 还是 fallback solver。

**面试官想听的点：**  
知道桥接层的角色：解耦、兼容、回退。

**常见踩坑：**  
说成“只是为了方便启动命令”。

---

### Q37. 求解器和 AI 工作流之间是什么关系？
**参考答案：**  
求解器在我的系统里既是验证器，也是信号源。对于 AI 来说，它不是附属功能，而是决策闭环的一部分。generator 修改关卡后，系统会重新做 solve 和 score，critic 也会基于这些确定性结果评审 candidate，controller 最终是否接受也会依赖这些结果。也就是说，求解器不是最后才调用一次，而是整个 Agent workflow 中的核心工具之一。

**面试官想听的点：**  
知道 solver 不只是“外挂算法”，而是 Agent 闭环里的基础设施。

**常见踩坑：**  
把 solver 和 AI 讲成两个平行系统，没讲清耦合点。

---

## 六、技术债、稳定性与生产化（7 题）

### Q38. 你这个项目现在最大的技术债是什么？
**参考答案：**  
我认为最大的技术债有三类。第一，规则逻辑在前端、Node 和 Java 里有多处实现，存在语义漂移风险。第二，前端编辑器在原型阶段把很多交互和状态逻辑集中在单一大文件里，维护成本偏高。第三，后台 modify job 和 memory 设计已经可用，但更偏单机原型，还不是完整生产级基础设施。它们都不是我没意识到的问题，而是阶段性取舍：我先把系统闭环跑通，再逐步收敛规则和基础设施。

**面试官想听的点：**  
诚实；知道技术债是什么、为什么形成、怎么还。

**常见踩坑：**  
说“没什么技术债”；或者把技术债讲成自己失误，而不是阶段性取舍。

---

### Q39. 工具调用失败或者模型幻觉怎么办？
**参考答案：**  
我会把问题拆成两层：工具失败和模型幻觉。工具失败时，系统会显式把错误返回给 agent，而不是假装成功；然后根据失败类型决定重试、降级还是停止。模型幻觉的问题，我尽量不让它发生在高风险环节：像合法性、可解性、最短步数、难度评估，这些都必须依赖确定性工具，而不是模型自由发挥。我的核心设计不是“让模型永不犯错”，而是“即使模型犯错，系统也能把错误限制在边界内”。

**面试官想听的点：**  
受控执行、失败分类、系统边界。

**常见踩坑：**  
只回答“多试几次”“改 prompt”。

---

### Q40. 你的异步 modify job 现在有什么局限？
**参考答案：**  
当前的 modify job 更像单机可运行的后台任务模型，而不是生产级任务系统。它的优点是简单、联调快；缺点是 durability 不够，比如进程重启后任务状态不一定保得住，多实例部署也不方便。这是我明确知道的一笔技术债。如果要继续往产品化走，我会把任务状态持久化，并引入外部 worker 或队列系统，把异步执行从进程内线程模型升级出去。

**面试官想听的点：**  
知道单机后台线程和持久化队列的差异。

**常见踩坑：**  
硬把 demo 级方案说成“生产可直接用”。

---

### Q41. 你怎么做评估，知道 Agent 改动有没有变好？
**参考答案：**  
我会把评估拆成确定性指标和用户体验指标。确定性指标包括是否合法、是否可解、最短步数、分支、冗余对象等，这些可以通过 validate/solve/score 直接得到。用户体验指标则包括是否更符合 brief、修改是否够小、解释是否清晰，这部分由 critic report 和人工抽样共同补充。也就是说，我不会只看模型输出文本“感觉更好了”，而是一定要有 deterministic metrics 作为地基。

**面试官想听的点：**  
知道 Agent 评估不能只靠主观感受。

**常见踩坑：**  
把评估回答成“看用户喜不喜欢”。

---

### Q42. 如果继续做生产化，你会怎么增强可观测性？
**参考答案：**  
我会围绕 request、session、tool、job 四个层次做可观测。request 层跟踪一次用户请求从前端到后端的完整链路；session 层看多轮会话如何演进；tool 层记录每次工具调用的成功率、耗时和错误类型；job 层跟踪后台 modify job 的阶段、重试和最终状态。对 Agent 系统来说，可观测性不是可选项，因为没有 trace 和 tool history，就很难解释“为什么这轮改关失败了”。

**面试官想听的点：**  
分层可观测意识；知道 Agent 系统为什么更依赖 trace。

**常见踩坑：**  
只说“打日志”。

---

### Q43. 如果面试官质疑你项目“不够成熟”，你怎么回答？
**参考答案：**  
我会承认它不是一个全部产品化完成的系统，但它已经有很清晰的工程骨架，而且关键问题我不是没想到，而是有意识地分阶段落地。比如我先把规则、求解器、AI workflow、原子编辑和 session memory 这些最核心的架构问题打通，再逐步处理规则收敛、任务基础设施和更强的评估体系。也就是说，它不是“功能堆出来的 demo”，而是一个已经具备演进方向的系统。

**面试官想听的点：**  
成熟的工程判断；知道原型和产品的边界。

**常见踩坑：**  
急着辩解“其实已经很成熟了”，反而失分。

---

### Q44. 如果让你总结这个项目的工程哲学，你会怎么说？
**参考答案：**  
我会说，这个项目的核心不是“让模型更自由”，而是“让模型在受约束的系统里更有用”。所以我优先做分层、结构化中间契约、工具调用、原子编辑、确定性求解、RAG grounding、session memory 和 controller 决策。换句话说，我希望它是一个可控、可解释、可回放的 Agent 系统，而不是一个只会生成文本的黑盒。

**面试官想听的点：**  
有自己的设计原则；能从细节上升到方法论。

**常见踩坑：**  
只谈技术点，不谈设计哲学。

## 附：面试前速记版结论
- 项目定位：不是编辑器叠 AI，而是“谜题编辑 + 求解器 + 受控 Agent workflow”。

- RAG 核心价值：规则 grounding + 案例 grounding。
- Memory 核心价值：保存任务状态，不只是聊天历史。
- Solver 核心价值：让 AI 结果站在确定性规则上，而不是只靠模型主观判断。
- 技术债核心：规则多实现并存、前端单文件过大、后台任务仍偏单机。
